## Decisions

### D1. Mode picker = entry modal, not step reorder (M1.2)

**Tension:** PRD §S1 says "step-2 of campaign creation gets a clearer mode selector". 2026-05-05 sync says "modal asking users to choose … before platform selection". Current flow has the toggle at step 5.

**Council resolution:** ship a `<Dialog>` that pops once on entry to `/creator/campaigns/new`, asking the creator to pick Clip or Subscriber with "best-for" framing copy. Sets `state.payoutType` + resets `state.platforms` (since per-view and per-sub have different platform options). Editing an existing draft skips the modal. The step-5 payoutType toggle stays so users can flip their mind later.

**Why this over a step renumber:** the value of the transcript decision is "force the choice early". Reordering 5+ steps would cascade through `canNext`, `getFirstIncompleteStep`, scrolling refs, and step-render branches — high disruption, low UX delta over a modal. Modal lands the same UX outcome with one new dialog and zero state-machine surgery.

### D2. Platforms in step 4 are mode-aware (M1.3)

Per-view selector: TikTok + Instagram (already aligned with M0.4 YouTube cleanup).
Per-sub selector: Instagram, TikTok, Reddit, X (per 2026-05-05 sync).

Implementation: ternary on `state.payoutType` inside the step-4 render. When the modal flips mode, `state.platforms` is cleared so leftover selections from the other mode don't poison the multi-select.

Frontend `Platform` type extended to `tiktok | instagram | youtube | reddit | x`. Backend `Platform` type (`server/src/scrapers/scrapers.types.ts`) intentionally unchanged — Reddit/X are tracking-link tags only, no scraping path.

### D3. Trust filter is page-local, not server-side (M1.6)

Trust score is computed live from `payout_events` (no stored field). Adding a server-side trust filter to `campaigns.list()` would duplicate the trust-sort SQL clauses (already 14 lines of SELECT-COUNT subqueries) and balloon scope. For v1.2, the hub trust filter ("100% paid track record only") filters the **visible page** in JavaScript using `creator.trust.allTime.creatorPaidRate`.

**Trade-off:** if the user paginates past page 1, trust filter resets effective on each page (since server returned a fresh 20). Acceptable for launch; revisit if hub becomes a primary funnel.

### D4. Card shows trust + budget remaining + rate + status; payout methods deferred (M1.4)

PRD §S1 says subscriber cards should show: payout per sub, payout methods, creator trust, budget remaining, status.

Implemented: payout per sub (top-right rate chip now reads `$X / sub` for per-sub), creator trust (TrustBadge — already present), budget remaining (`$X left` — already present), status (banding). Type badge (top-left) is new.

**Not implemented:** payout methods on the card. Adding 3+ method icons + the existing platform icons + trust + rate would visually overwhelm a card already at density. Methods stay on the detail page (where they already show). If user research surfaces a need for at-a-glance method matching, we'll revisit.

### D5. Login + hub copy refresh (M1.7, M1.8)

- Hub: description copy now mentions both modes ("clip campaigns" + "subscriber campaigns" or "earn per 1,000 views or per acquired subscriber").
- Login hero: "growth marketplace" replaces "clipping marketplace"; body copy mentions both modes; feature row "Performance-based payouts" rewritten to call out both rates; multi-platform feature row mentions Reddit + X for sub links.

### D6. Filter sheet ordering (M1.5)

Order in the sheet: **Campaign type → Trust → Platform → Sort → Min reward**. Type and trust come first because they coarse-narrow the result set; platform / sort / minRate refine within. Active filter count includes both new filters.
