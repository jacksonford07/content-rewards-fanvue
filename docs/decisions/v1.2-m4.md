## Decisions

### D1. Two SDKs, two key envs

`posthog-js` on the frontend (`VITE_POSTHOG_KEY` / `VITE_POSTHOG_HOST`) and `posthog-node` on the backend (`POSTHOG_KEY` / `POSTHOG_HOST`). Both gated — when keys are unset, every method no-ops so the code can ship before the keys do.

Per PRD §S5 / 2026-05-05 sync, Content Rewards runs on its own PostHog account separate from any Fanvue-wide account. No cross-product identify. **Distinct ID = `users.id`** (the app's user UUID).

### D2. Auto-capture OFF (frontend) + manual events only

PRD §S5: "Manual events only — auto-capture off." `posthog.init` configured with `autocapture: false`, `capture_pageview: false`, `capture_pageleave: false`. Every event is fired explicitly through `track()` (frontend) or `posthog.capture()` (backend).

### D3. Most events fire backend-side; only one frontend event

Backend has the full state for every event in PRD §S5 — campaign creation, submission lifecycle, mint, revoke, mark-paid, payout confirmation. The single frontend-only event is `tracking_link_copied` (we can't observe a clipboard write from the server).

### D4. NestJS @Global() module for PostHog

Single shared `PosthogService` provider. Made `@Global()` so feature modules (campaigns, submissions, etc.) can inject it without per-module wiring. One client instance for the whole app; `OnModuleDestroy` flushes pending events on shutdown.

### D5. Funnel events first; secondary events deferred to M4.x

This PR wires **7 events** (6 server, 1 client) covering the main funnel:

| Event | Where | Reason |
|---|---|---|
| `campaign_created` | `campaigns.service.create()` | Funnel start |
| `subscriber_campaign_created` | Same site, gated | Subscriber-mode adoption |
| `submission_created` | `submissions.service.apply()` | Apply flow |
| `tracking_link_minted` | `tryMintTrackingLink` | Mint success |
| `tracking_link_copied` | Frontend `PerSubClipperPanel` | Promoter shared the link |
| `tracking_link_revoked` | `submissions.service.revoke()` | Creator stopped accrual |
| `submission_marked_paid` | `submissions.service.markPaid()` | Payout flow start |

**Deferred to a follow-up M4.1.x PR** (PRD §S5 lists them but they'd inflate this PR's review surface):

| Event | Where it lives | Why deferred |
|---|---|---|
| `payment_confirmed` | `confirmPayout()` + cron auto-confirm | Trust funnel — useful but not gating |
| `payment_disputed` | `disputePayout()` | Lower volume |
| `dispute_resolved` | Admin resolve | Admin-only |
| `payout_settings_saved` | Clipper payout method save | Settings event |
| `oauth_scope_upgrade_completed` | Auth callback | One-shot per creator |
| `subscriber_attribution_synced` | `cron.service.syncSubscriberAttribution` | Highest volume — wire after dashboards exist so we can size the data load |

The deferred events are non-blocking for the main funnel dashboards. Wire them in a follow-up after smoke-testing the funnel.

### D6. Dashboards (M4.2) are setup work, not code

Per PRD §S5 the dashboards are: subscriber-campaign funnel, click → acquired-sub conversion, mark-paid → confirm conversion, dispute rate, payment completion by method.

These are PostHog UI configurations — Insights / Funnels / Trends. They get built in the PostHog dashboard once events flow. **Setup doc `docs/setup/v1.2-m4.md` lists each dashboard with the events to use and the breakdowns to set.** No code change.

### D7. Bundle size impact

`posthog-js` adds ~50KB gzipped to the frontend bundle. Already over the chunk warning threshold; this push it slightly higher. Acceptable for v1.2 — bundle-splitting is an M5+ concern, not analytics-blocking.
