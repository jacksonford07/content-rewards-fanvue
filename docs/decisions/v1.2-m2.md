## Decisions

### D1. Schema: two columns, one migration (M2.1)

`campaigns.traffic_rules` (text, nullable) + `submissions.last_clicks` (int, default 0) added in `0006_v1_2_m2_workspace.sql`. Both nullable / defaulted so the migration is non-blocking on existing rows. Drizzle journal entry incremented to idx 6.

### D2. Clicks ingestion does not accrue earnings (M2.2)

The cron snapshots `link.clicks` alongside `link.engagement.acquiredSubscribers` and persists both deltas. Clicks-only deltas trigger an update so the workspace + roster show fresh click counts even when subs haven't moved. **But:** earnings still accrue on subs only — per-sub rate is on subscribers, never clicks. This avoids accidentally double-paying when click counts spike.

Cron loop now skips a row when **both** sub-delta and click-delta are zero (was: skip on sub-delta zero only).

### D3. Sync cadence 6h → 30 min (M2.3)

`render.yaml` schedule for `sync-subscribers` updated from `0 */6 * * *` to `*/30 * * * *`. Each cron run does at most one `GET /tracking-links` per creator (already grouped). Fanvue rate limits are unconfirmed (PRD §15 Q3) — assuming fair-use; will monitor 429s after deploy.

### D4. Tracking link name format (M2.7)

New name: `<campaign title> · Clipper N (@handle)` where N = count of approved+ submissions on this campaign at mint time + 1, **excluding** the row being minted (so retries don't double-increment).

Status set used for the count: `approved | auto_approved | ready_to_pay | paid_off_platform | disputed`. Excludes `pending` and `rejected` so manual-mode applies that haven't been approved yet don't bump N for the next clipper.

### D5. Roster lives on budget page, not detail page (M2.8 alternative)

PRD §S4 placed the roster on `campaign-detail-page`. **Council resolution:** `campaign-budget-page` already has a per-sub clipper roster (4 sections — pending / accruing / ready / settled — with approve/reject/mark-paid actions). Duplicating it on the detail page would mean two surfaces showing the same data with subtly different actions.

Instead: add a **"Manage promoters" CTA** on `campaign-detail-page` (owner + per-sub only) that routes to `/creator/campaigns/{id}/budget`. Roster on the budget page gains a Clicks column. Owners reach the same outcome (see all promoters, take actions) with one less duplicated surface.

Trade-off: creator must navigate. Acceptable given the budget page is already the actions hub for per-sub campaigns.

### D6. Anti-bot ack is page-local (M2.11)

Checkbox required to enable the Apply button in `PerSubClipperPanel`. State is local to the component — **not persisted** on the submission. Adding a `submissions.acknowledged_anti_bot_at` column was considered but:
- The check is enforced before apply succeeds, so a "pre-apply" timestamp is redundant.
- Future audit / dispute work might want it; defer until that need is concrete.

If we later need an auditable record, add the column + set it inside `submissions.service.ts apply()`.

### D7. Markdown rules render as plain text (M2.4)

Traffic rules render with `whitespace-pre-wrap` (preserves newlines, no markdown parsing). This avoids adding `react-markdown` (~50KB gz) for what's typically a 3–6 line policy snippet. If creators start asking for bold/lists, swap in a markdown component then.
