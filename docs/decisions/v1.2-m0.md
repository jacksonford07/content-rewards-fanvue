## Decisions

### D1. B1 breadcrumbs — wired on inbox + my-submissions pages

Added `<Breadcrumb>` block above `<PageHeader>` on `creator-inbox-page.tsx` (My campaigns → Submission inbox) and `my-submissions-page.tsx` (Campaigns → My submissions). Mirrors the pattern already used on `campaign-budget-page.tsx`.

### D2. OAuth scope-upgrade redirect — set `cr_post_login_redirect` before bounce

The auth callback at `auth-callback-page.tsx` already honours `localStorage.cr_post_login_redirect` for any user whose role is `clipper` or `creator` — the bug was that `create-campaign-page.tsx` never set the key before the scope-upgrade redirect.

Fix: before `window.location.href = …/auth/fanvue?scope=tracking_links`, save the campaign as draft (already happened) and write `localStorage.setItem("cr_post_login_redirect", "/creator/campaigns/{draftId}/edit")`. After re-auth the user lands back on the draft they were editing.

### D3. Silent mint failure — defer creator-facing toast to M3

`submissions.service.ts:1023–1056` silently logs and skips mint when `write:tracking_links` is missing. PRD §S7 calls for "defensive check at mint time + retry path." The retry path is more naturally part of the revoke / mint surface in M3 (which already touches the link lifecycle). For M0, leaving the silent-skip behaviour intact and noting it here as **deferred to M3.5** (a follow-up ticket I'll add to the tracker after merge if it's not already on the roster).

### D4. YouTube removal — frontend only; keep backend enum + scrapers

**Council recommendation accepted.** Per-view = TikTok + Instagram only going forward, but historical YouTube submissions (in-flight scrapes, pending payouts on existing per-view campaigns that whitelisted YouTube) must continue to process. Therefore:

**Removed (this branch):**
- `create-campaign-page.tsx:612` — YouTube dropped from per-view platform selector
- `hub-page.tsx:47` — YouTube dropped from filter options
- `login-page.tsx:79–80, 103` — marketing copy reads "TikTok and Instagram Reels"

**Kept (intentional):**
- `Platform` type union still includes `"youtube"` (`src/lib/types.ts:3`)
- `PlatformIcon` YouTube branch reachable for historical data
- Backend scrapers (`youtube-scraper.service.ts`, `youtube-downloader.service.ts`)
- URL parser detection (`scrapers.types.ts:18`, `campaign-detail-page.tsx:113`)
- Seed data — left as is, will be regenerated as needed

**Backend create-time guard — declined for M0.** No frontend path exposes YouTube anymore. Adding a `BadRequestException` on `payoutType="per_1k_views" && allowedPlatforms.includes("youtube")` is defence-in-depth but introduces a foot-gun for any historical update flow. If we later see API-direct attempts to whitelist YouTube on a new per-view campaign, fold it in then.
