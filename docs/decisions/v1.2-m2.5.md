## Decision: defer entirely until Fanvue confirms the endpoint

The blocker (M2.5.1) is unanswered: **which Fanvue API endpoint exposes a creator's current subscription price (free vs paid + amount), and what scope does it require?** Without that answer, every downstream choice is speculative:

- **Data type for the column.** Is the price denominated in USD cents, multi-currency, an enum? We don't know.
- **Snapshot semantics.** Does Fanvue rotate the price over time? If a creator changes from $9.99/mo to $14.99/mo mid-campaign, does the API return the value at the time of fetch or has it always been current? Determines whether we cache once at creation or refetch.
- **Free vs paid encoding.** Is "free page" represented as `null`, `0`, or a separate boolean? We can't future-proof the schema without seeing the response.
- **OAuth scope.** May require a new scope (`read:profile`?) to fold into the existing tracking-links upgrade flow. Until known, we can't ship the UI guidance.

## Why I'm not shipping speculative scaffolding

Tempting to add `campaigns.creator_sub_price_at_creation_cents` (int, nullable) and call it half-done. **Declined.** Reasons:

1. The system-prompt guidance is explicit: don't add infrastructure for hypothetical future requirements. The column would sit unused, and once we hear back from Fanvue we'd likely change the shape (currency? snapshot timestamp? free-page boolean?). Schema changes are easier to design once than to evolve incrementally.
2. A feature-flagged UI ("show this label if `VITE_FANVUE_SUB_PRICE_ENABLED`") is a mock pretending to be done work. Worse, the placeholder copy would have to be invented without seeing the data.
3. The actual ticket asks for **decision support at campaign creation**. Without a real number to display, the feature has zero user value.

## What I did instead

- Recorded the decision here.
- Left M2.5.1 (the API question) as a tracker item to nudge Fanvue on.
- The schema, UI, and copy land **after** Fanvue confirms — in a follow-up PR named `v1.2-m2.5-impl` or similar.

## Re-entry checklist (when the answer arrives)

When the Fanvue API team confirms the endpoint:

1. Update tracker ticket M2.5.1 with the endpoint, scope, response shape.
2. Branch `v1.2-m2.5-impl` from main.

4. **Backend**: campaigns service `create()` fetches the price via Fanvue at campaign create time and persists. If the endpoint requires a new scope, fold it into the existing tracking-links scope-upgrade prompt (M0.3 redirect path already handles draft restoration).

   - Free: "Your page: **Free** — you're paying for traffic, not direct revenue. Make sure post-subscription monetisation justifies the rate."
6. Re-open this decision file and convert "DEFERRED" to a real decision log.
