# v1.2 M3 — Revoke Link UX Decisions

**Branch:** `v1.2-m3`
**Tickets:** M3.1, M3.2, M3.3, M3.4
**Run mode:** autonomous (council = parallel subagent review)
**Date:** 2026-05-05

---

## Decisions

### D1. Distinct `revoked` status, not reused `rejected` (M3.3)

**Council recommendation accepted from CC2.** Adding `revoked` to the `submissions.status` enum preserves the audit trail: rejected = "we never let this clipper accrue", revoked = "we approved them, they accrued, we cut them off mid-flight". Different stakeholder questions, different answers, different states.

Costs: one new value in the Drizzle text enum + one branch on every status-mapping switch (creator-budget-page, my-submissions-page, etc.). Cheap.

**No migration file.** `text("status", { enum: [...] })` is a Postgres `text` column with a TypeScript-only constraint; existing rows keep their values, new `revoked` writes succeed. `db:push` with the new schema reconciles without an ALTER. Verified by inspection — same pattern as M2.10's status enum extension on the campaigns table.

### D2. Revoke ≠ Ban (M3.1)

`ban()` already exists and is heavier — also inserts into `campaign_bans` and zeros `pendingEarningsCents`. Revoke is intentionally lighter:
- Deletes the Fanvue tracking link.
- Sets `status = "revoked"`.
- **Leaves accrued earnings intact** so the creator can mark-paid the prior accrual.
- Does NOT ban the clipper from this campaign or the platform.

This matches the CC2 decision log: "Revoke behaviour: deleting the Fanvue link stops further accrual on that submission; prior accrued amount stands."

If a creator wants to also blacklist a misbehaving clipper, they hit Ban, not Revoke. Two separate buttons; two separate intents.

### D3. Best-effort Fanvue delete on revoke (M3.1)

If the Fanvue DELETE call fails (rate-limit, expired token, link already gone), revoke still flips the local status. The user-visible state of "this promoter no longer accrues" is what matters; the orphaned Fanvue link will get cleaned up by Fanvue eventually or noticed on the next sync (M3.4 auto-revoke would re-trigger as a no-op since status is already `revoked` and the cron skips by status filter anyway).

### D4. Auto-revoke on `listAllLinks` miss (M3.4)

Cron already calls `listAllLinks` once per creator per run and indexes by uuid. Existing behaviour: if the list missed a known `trackingLinkUuid`, the loop `continue`'d silently. New behaviour: flip the submission to `revoked` and log it.

This catches:
- Creator deleted the link manually from the Fanvue dashboard (the scamming-attempt scenario from the 2026-05-05 sync).
- Fanvue rotated the link or had a data issue.

**Not caught:** transient API errors. The cron's outer try/catch logs and continues, so a single failed `listAllLinks` call doesn't sweep an active campaign's submissions. Fine — a one-cycle false positive would be visible in logs as `Auto-revoked submission … not in Fanvue listing` for many submissions at once, which would be a clearly diagnosable Fanvue outage rather than a real revocation pattern.

### D5. New FanvueNotFoundError class (M3.1 / M3.4)

Added 404 handling in `FanvueTrackingLinksService.fetch()` that throws `FanvueNotFoundError`. Currently no caller specifically catches it — `listAllLinks` doesn't 404 (it returns a list), and the existing `deleteLink` is best-effort. The class exists so future per-link reads (if Fanvue adds a `GET /tracking-links/:id` we want to use) can surface a typed error instead of a generic API error.

### D6. Revoke UI lives on accruing rows only

The revoke button shows on the **Accruing** section of the budget page roster — not Ready, Settled, or Pending. Why:
- Pending = link not minted yet; nothing to revoke.
- Ready / Settled = sweep already happened or payout in progress; revoking would confuse the payout flow.
- Accruing = the only state where stopping accrual changes outcomes.

Revoked rows render in a new `Revoked` group at the bottom of the roster (read-only).
