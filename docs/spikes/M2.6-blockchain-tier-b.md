# M2.6 spike — Blockchain Tier B feasibility

**Question:** how much engineering effort is required to verify a crypto
payout *on-chain* (Tier B) versus the manual tx-hash + explorer link we
ship in M2.5 (Tier A)? Should it land inside M2 or defer to v1.2?

## Summary

| Tier | Effort | Trust signal | Recommendation |
|---|---|---|---|
| **A — manual hash** (shipped in M2.5) | done | weak (clickable explorer link, easily faked) | shipped |
| **B — on-chain verify** (this spike) | **5–7 days** for ETH/USDC + Solana/USDC; +2 days for native BTC | strong (matches recipient address, amount, token, confirmations) | **Defer to v1.2** |

## Recommendation

**Defer Tier B to v1.2.** Reasoning:

- v1 launch volume is too low to justify the engineering. CC1 anticipates a
  small-cohort launch — likely <50 crypto payouts in the first month.
- Tier B's value is *fraud detection*. At <50 payouts/mo, fraud risk is
  bounded by a known reviewer (Jackson) eyeballing the dispute queue (M3.6).
  Manual hash + explorer link gets us most of the deterrent value at zero
  marginal cost.
- The dispute admin queue lands in M3.6 anyway. If a clipper disputes a
  payout, a human reviews the explorer link in <30 seconds. That covers the
  fraud surface Tier B would automate.
- 5–7 days of M2 → M3 critical-path schedule cost outweighs the protection,
  and the implementation tightens around RPC providers, retry logic, USDC
  contract addresses, and reorg handling — all things best done with real
  payout data informing the design.

A v1.2 follow-up ticket should pick this up after the first 30 crypto
payouts have landed, when we have:
- A real distribution of methods to know which chains to support
- Sample dispute volume to calibrate "is this worth it"
- An RPC budget signal (free tiers may already cover us indefinitely)

## Implementation sketch (for the v1.2 ticket)

### What "verified" means per chain

| Chain | Source of truth | Recipient match | Amount match | Token contract | Confirmations |
|---|---|---|---|---|---|
| **ETH** | `eth_getTransactionByHash` + `eth_getTransactionReceipt` | `tx.to == saved` | `tx.value == amountCents/100 in wei` | n/a (native) | ≥ 3 blocks (~36s) |
| **USDC on ETH** | `eth_getTransactionReceipt` + log decode | `Transfer.to == saved` | `Transfer.value == amount * 1e6` | `tx.to == 0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48` (USDC) | ≥ 3 blocks |
| **Solana** | `getTransaction(sig, "confirmed")` | post-balance change on saved address | balance delta == amount lamports | n/a (native) | confirmed (1 cluster vote) |
| **USDC on Solana** | `getTransaction(sig, "confirmed")` + token-balance delta | token-balance change on saved address's USDC ATA | delta == amount * 1e6 | mint == `EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v` | confirmed |
| **BTC** | mempool.space REST or local node | output to saved address | `vout.value == amountCents * 100` (sats) | n/a | ≥ 3 confirmations |

### RPC provider options

| Provider | Free tier | Notes |
|---|---|---|
| Alchemy | 300M CU/mo (≈ ~10K reads) | Multi-chain (ETH, Solana). Best DX. Recommended for ETH side. |
| Helius | 1M credits/mo on free | Solana-native. Best for Solana side. |
| QuickNode | 10M credits/mo | Multi-chain incl. BTC. Backup. |
| Public RPCs | unmetered | Rate-limited and unreliable. Only for fallback. |
| mempool.space | unmetered REST | BTC-only; perfect for the BTC path. |

**Recommendation:** Alchemy (ETH/USDC-ETH) + Helius (Solana/USDC-Sol) +
mempool.space (BTC). All free-tier-compatible at v1 volumes.

### Schema additions

```sql
ALTER TABLE payout_events
  ADD COLUMN chain_verified boolean DEFAULT false NOT NULL,
  ADD COLUMN chain_verified_at timestamp with time zone,
  ADD COLUMN chain_verification_error text;
```

### Service shape

```ts
// server/src/payout-events/blockchain-verifier.ts
interface VerifyResult {
  verified: boolean
  error?: string  // null when verified=true
}

export interface BlockchainVerifier {
  verify(args: {
    method: PayoutMethod
    txHash: string
    expectedRecipient: string
    expectedAmountCents: number
  }): Promise<VerifyResult>
}
```

Run on a queue (BullMQ or pg-boss) — first attempt 30s after mark-paid,
retry with exponential backoff up to 24h (long tail for slow finalisations
or wallet propagation).

### Trust score interaction (M3)

Tier B verified payouts could weight the trust score higher than self-
reported ones. Suggested: in `confirmed_payouts / verified_submissions`,
chain_verified payouts count as 1.0; manual-only as 0.7. Cap visible
score impact so creators using non-crypto rails aren't penalised.

### Risk items
- **Reorg handling**: ETH reorgs are rare past 3 confirmations but possible.
  If a verified tx gets reorged out, the payout_event row needs to flip
  back to unverified. Solution: schedule a re-check 1h post-verify; if it
  fails, revert.
- **Wallet rotation**: clippers might change their saved wallet between
  the creator copying it and sending the tx. We snapshot
  `valueSnapshot` at mark-paid time, so the verifier compares against the
  snapshot, not the current saved value. This is correct.
- **USDC contract drift**: the canonical USDC contract addresses on
  ETH/Solana are stable, but should be configurable env vars in case Circle
  ever migrates.
