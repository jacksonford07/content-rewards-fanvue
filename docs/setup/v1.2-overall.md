# v1.2 — Overall Setup Guide

A single index of everything Iniaki needs to do to get v1.2 fully running, in the order it should happen. Per-milestone setup docs (`docs/setup/v1.2-m{N}.md`) have the granular detail; this doc is the linear "do these in this sequence" checklist.

---

## Pre-merge

- [ ] **Smoke against the integrated branch.** Use `docs/smoke/v1.2.md` (mass smoke test). Vercel preview = whatever's deployed for `v1.2-cross-cutting` / PR #19, but in this repo Vercel only deploys main, so this stage is "review code + read decision docs" rather than "click around a preview".

---

## Merge step

- [ ] **Merge PR #19** (`v1.2-cross-cutting → main`). Single merge commit; rollup of all seven milestones. Triggers Vercel + Render production redeploys.
- [ ] After merge, the stacked PRs (#13–#18) auto-close.

---

## Database migration (M2 only — required before Render finishes redeploying, otherwise the cron will crash)

- [ ] Run against prod Neon Postgres:
  ```bash
  cd server
  DATABASE_URL="<prod neon url>" pnpm db:push
  ```
- [ ] Confirms two new columns exist:
  - `campaigns.traffic_rules` (text, nullable)
  - `submissions.last_clicks` (int, default 0)
- [ ] If `db:push` errors, the safe rollback is reverting PR #19's merge commit on `main` and re-deploying main. The migration is idempotent so re-running after revert is fine.

**Note on M3:** the new `revoked` value on the `submissions.status` enum is TS-only (text column, no Postgres native enum). `db:push` reconciles it with no SQL change — same migration run covers both M2 and M3.

---

## Render Blueprint resync (M2 cron cadence + M2 secret check)

- [ ] Render dashboard → Blueprints → existing instance for `content-rewards-fanvue` → **Sync**.
- [ ] Confirm `sync-subscribers` cron service shows schedule `*/30 * * * *` (was `0 */6 * * *`).
- [ ] Confirm `CRON_SECRET` is still set on the `content-rewards-fanvue` web service. Crons inherit it via `fromService.envVarKey`.
- [ ] If Blueprint sync errors out, fallback: edit the cron service directly in the Render UI (Settings → Schedule → `*/30 * * * *`).

---

## PostHog setup (M4 — events no-op until env vars set, so deploy is safe whatever order)

- [ ] Create a Content Rewards project in https://us.posthog.com (separate from any Fanvue-wide project — no cross-product identify per PRD §S5).
- [ ] Note the project's Public API key.
- [ ] **Vercel** (Settings → Environment Variables; set on Production + Preview + Development; redeploy):
  ```
  VITE_POSTHOG_KEY=phc_xxxxxxxx
  VITE_POSTHOG_HOST=https://us.i.posthog.com
  ```
- [ ] **Render** (`content-rewards-fanvue` web service → Environment):
  ```
  POSTHOG_KEY=phc_xxxxxxxx           # same public key
  POSTHOG_HOST=https://us.i.posthog.com
  ```
- [ ] Build the **Subscriber Campaign Funnel** dashboard in PostHog UI (steps in `docs/setup/v1.2-m4.md`).

## OAuth scope-upgrade verification (M0 — once-off check after deploy)

- [ ] Pick a creator account that does NOT have `write:tracking_links` granted yet (or revoke it from the Fanvue dev portal).
- [ ] Start a per-sub campaign as them; publish.
- [ ] Confirm: bounce to Fanvue OAuth → grant scope → land back on `/creator/campaigns/{draftId}/edit` (not `/creator/campaigns`).

## After all of the above

- [ ] Walk `docs/smoke/v1.2.md` end-to-end against prod.
- [ ] Watch Render logs for the first `sync-subscribers` tick after deploy: should log `updated: N` (N ≥ 0). Click counts should populate `submissions.lastClicks` for any active per-sub submissions.
- [ ] Watch PostHog Live Events: `campaign_created`, `submission_created`, `tracking_link_minted`, etc. should land within seconds of triggering each.

---

## If something goes wrong

- Rollback for the whole rollup: `git revert -m 1 <merge-commit-sha>` on `main`, then `git push origin main`. PR #19 is the merge commit reference.
- Per-milestone rollback isn't supported because the merge is one commit. If a single milestone is broken, fix forward on a new branch from main.
